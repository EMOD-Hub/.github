# Contributing to EMOD-Hub Projects

Welcome! We are glad you are interested in contributing to EMOD-Hub projects.

Projects in this organization are provided as open source software for community
use and development under the MIT license.

**Unless otherwise noted, these projects are no longer actively maintained or supported
by IDM or the Gates Foundation.**

Community contributions are welcome. Trusted collaborators may review and merge pull
requests, but no guarantees are made regarding support, pull request review,
security response, maintenance, or release timelines.

Please review the guidelines below before opening issues or submitting pull requests.

## Getting started

Before contributing, please review the documentation at:

* https://emod.idmod.org

Repository-specific setup instructions, development workflows, and additional
documentation can usually be found in the repository README.

## Discussions

Have a comment or a question? Check out our [Discussions](https://github.com/orgs/EMOD-Hub/discussions) space.

## Ways to contribute

We welcome the following types of contributions:

1. [Issues](https://github.com/EMOD-Hub/issues-and-discussions/issues)

   * Bug reports
   * Feature requests

2. Pull requests

   * Bug fixes
   * Tests reproducing an issue
   * Documentation improvements
   * New features or enhancements

Project [discussions](https://github.com/orgs/EMOD-Hub/discussions),
issue tracking, and contribution coordination take place on GitHub.

## Request new features or report bugs

If you encounter a bug, unexpected behavior, missing documentation, or would like to propose an enhancement, please first review the relevant repository documentation and existing issues.

Before opening a new issue:

1. Review the repository README and available documentation at https://emod.idmod.org.
2. Search existing [issues](https://github.com/EMOD-Hub/issues-and-discussions/issues)
and [discussions](https://github.com/orgs/EMOD-Hub/discussions) to see whether the
topic has already been raised.
3. If an existing issue matches your feedback, consider adding a 👍 reaction instead of a new comment unless you have additional useful information to share.

When opening a new issue, please provide as much relevant detail as possible, including:

- a clear and descriptive title
- whether you are reporting a bug or requesting an enhancement
- your use case or the problem you are trying to solve
- steps to reproduce the issue, if applicable
- expected and observed behavior
- error messages, logs, screenshots, or code snippets
- relevant version information
- links to related files, projects, or discussions, if useful

Community members and collaborators may respond to issues and questions, but response times and resolution are not guaranteed.

## Submit a pull request

To contribute code or documentation changes:

1. Fork and clone the relevant repository.

2. Follow the repository-specific installation and development instructions provided
in the README or documentation.

3. Create a new branch:

   ```
   git checkout -b my-branch-name
   ```

4. Make your changes, including clear and descriptive commit messages.

5. Push your branch to your fork:

   ```
   git push origin my-branch-name
   ```

6. Open a pull request against the main repository.

Pull requests are more likely to be reviewed and merged when they:

* are focused and self-contained
* are up to date with the main branch and free of merge conflicts
* address a specific bug, limitation, or enhancement
* follow the existing code style and project structure
* include relevant tests, documentation, or examples when appropriate
* include complete [Google-style docstrings](https://www.sphinx-doc.org/en/master/usage/extensions/example_google.html) and comments for new functionality
